## BK Vibe Coding Config

本项目启用 bk-vibe-coding，流程配置以项目根目录 bk-vibe-config.json 为准。

- TAPD_WORKSPACE_ID: <your_workspace_id>
- DEFAULT_BRANCH: master
- TEST_COMMAND: npm test
- PUBLISH_COMMAND: npm publish
- COMMIT_TEMPLATE: feat|fix|refactor|docs|chore: <subject> --story=<id>

## 任务流（必须串行执行）

```
阶段一            阶段二              阶段三                      阶段四              阶段五
需求沟通与确认 ──▶ HTML 原型生成 ──▶  PRD 撰写（含截图采集） ──▶  需求拆分        ──▶  研发闭环
     ↓                 ↓                    ↓                        ↓
  检查点确认        检查点确认          检查点确认               检查点确认
```

每个阶段的产出是下一阶段的输入，禁止跳过。

---

## 检查点确认机制（强制执行）

**每个阶段完成后，Agent 必须输出检查点清单，用户确认后方可进入下一阶段。**

### 检查点清单模板

```markdown
## 检查点确认 - 阶段X：[阶段名称]

### 本阶段产出
- [ ] 产出项 1：[具体产出物及路径]
- [ ] 产出项 2：[具体产出物及路径]

### 待确认事项
1. [需要用户确认的具体问题]
2. [需要用户确认的具体问题]

### 下一步操作
确认后将进入 [下一阶段名称]，执行 [具体操作]。

---
请回复「确认」继续，或提出需要修改的内容。
```

### 检查点行为规则

1. **必须等待用户明确回复**：用户回复「确认」「OK」「可以」等肯定词汇后才能继续
2. **支持部分修改**：用户可以针对清单中某一项提出修改意见，Agent 修改后重新输出检查点清单
3. **禁止跳过检查点**：即使用户主动要求跳过，也应提醒风险并建议完成确认
4. **自动衔接下一阶段**：用户确认后，Agent 应自动开始下一阶段的工作，无需用户再次触发

---

### 阶段一：需求沟通与确认

- **技能**：`product-design`（第一步）
- **输入**：用户的初始需求（设计稿 / 截图 / Figma 链接 / 口头描述 / 一句话想法）
- **产出**：`docs/{功能名}_需求大纲.md`（结构化需求大纲）
- **完成标准**：用户明确回复确认大纲可行

本阶段的目标是把模糊的需求变成清晰的、双方达成共识的需求大纲。**禁止在本阶段直接生成原型或文档。**

执行要点：

1. **理解意图** — 先分析用户提供的输入（截图/描述/链接），提炼出对需求的初步理解，向用户复述确认
2. **主动提问** — 围绕以下维度逐一引导用户补充信息，每轮聚焦 2-3 个问题，避免一次性抛出过多问题：
   - **业务背景**：这个功能要解决什么问题？现状是什么？
   - **目标用户**：谁会用这个功能？他们的角色和使用场景是什么？
   - **核心功能**：必须做的功能有哪些？哪些是锦上添花的？
   - **页面结构**：大致包含哪些页面/模块？主要的信息展示和操作是什么？
   - **关键交互**：有哪些重要的操作流程？有什么特殊的交互要求？
   - **数据与字段**：核心数据有哪些？需要展示哪些字段？
   - **边界与约束**：有什么技术限制、权限要求、性能要求？
3. **识别模糊点** — 对用户没有提到但对设计至关重要的部分，主动指出并给出建议选项（而不是等用户想到）
4. **输出需求大纲** — 信息收集充分后，输出结构化的需求大纲文档，包含：
   - 功能名称
   - 需求背景（一段话）
   - 目标用户及角色
   - 功能模块清单（附优先级 P0/P1/P2）
   - 每个模块的核心能力描述（2-3 句话）
   - 页面/模块结构（文字描述或简要列表）
   - 关键交互流程
   - 已知约束和边界
5. **确认大纲** — 将大纲完整展示给用户，明确询问"这个需求大纲是否可以作为原型设计的基础？"。用户确认后方可进入阶段二

沟通原则：
- 用户说得越少，越要主动引导和追问
- 用户提供截图/设计稿时，先基于截图分析出需求理解，再针对性追问缺失信息
- 避免假设，不确定的点直接问用户
- 每轮对话结束时总结当前已明确和待确认的内容

**阶段一检查点清单（必须输出）**：

```markdown
## 检查点确认 - 阶段一：需求沟通与确认

### 本阶段产出
- [ ] 需求大纲文档：`docs/{功能名}_需求大纲.md`

### 待确认事项
1. 需求大纲内容是否完整覆盖所有功能点？
2. 功能模块的优先级划分（P0/P1/P2）是否正确？
3. 已知约束和边界是否准确？

### 下一步操作
确认后将进入 **阶段二：HTML 原型生成**，基于需求大纲设计并生成可执行的 HTML 原型。

---
请回复「确认」继续，或提出需要修改的内容。
```

### 阶段二：生成可执行 HTML 原型

- **技能**：`product-design`（第二步）
- **输入**：阶段一确认的需求大纲
- **产出**：`prototypes/{功能名}.html`（单文件可运行）+ 功能结构图 + 操作流程图
- **完成标准**：用户在浏览器中预览原型并确认

执行要点：
1. 基于需求大纲，设计页面布局和交互方案
2. 生成单文件 HTML 原型（内联 CSS/JS，Mock 数据填充，核心交互可操作）
3. 引导用户在浏览器中预览，收集反馈并迭代
4. 用户确认后方可进入阶段三

**阶段二检查点清单（必须输出）**：

```markdown
## 检查点确认 - 阶段二：HTML 原型生成

### 本阶段产出
- [ ] HTML 原型文件：`prototypes/{功能名}.html`
- [ ] 功能结构图（Mermaid）
- [ ] 操作流程图（Mermaid）

### 待确认事项
1. 原型页面结构是否与需求大纲一致？
2. 核心交互（Tab 切换、弹窗、展开收起等）是否可正常操作？
3. Mock 数据是否符合真实业务场景？

### 下一步操作
确认后将进入 **阶段三：PRD 撰写**，基于原型撰写完整的产品需求文档。

---
请回复「确认」继续，或提出需要修改的内容。
```

### 阶段三：根据原型撰写 PRD

- **技能**：`product-design`（第三步）
- **输入**：阶段二确认的 HTML 原型
- **产出**：标准 PRD 文档 + 审核报告
- **完成标准**：PRD 通过自检清单，审核报告已输出

执行要点：
1. 基于已确认的 HTML 原型，提取页面结构、字段、交互和状态
2. 按 PRD 模板撰写完整文档（参考 `product-design/references/prd-template.md`）
3. 撰写到"页面原型"章节时，调用 `prototype-capture` 技能对原型截图并回填（截图内容由 PRD 中定义的功能模块和交互状态决定）
4. 逐条检查撰写规范（参考 `product-design/references/prd-writing-guide.md`）
5. 必须输出审核报告（参考 `product-design/references/review-report-template.md`）

串行约束：
- 如果用户直接要求写 PRD 但没有原型，先引导完成阶段二；如果连需求大纲都没有，先引导完成阶段一
- `prototype-capture` 是本阶段的子步骤，不是独立阶段

**阶段三检查点清单（必须输出）**：

```markdown
## 检查点确认 - 阶段三：PRD 撰写

### 本阶段产出
- [ ] PRD 文档：`docs/{功能名}_需求文档_v1.0.md`
- [ ] 审核报告：`docs/{功能名}_需求文档_v1.0_审核报告.md`
- [ ] 截图文件：`docs/screenshots/*.png`

### 待确认事项
1. PRD 各章节（背景、功能设计、交互说明等）是否完整？
2. 截图是否已正确回填到 PRD 对应章节？
3. 审核报告中的各项检查是否通过？

### 下一步操作
确认后将 **自动进入阶段四：需求拆分与 TAPD 单据创建**，执行以下操作：
1. 将 PRD、原型、审核报告上传到 iWiki
2. 创建 TAPD 父需求
3. 按功能模块拆分创建子需求
4. 在父需求添加评论引用 iWiki 链接

---
请回复「确认」继续，或提出需要修改的内容。
```

**自动衔接**：用户确认后，Agent 必须立即调用 `prd-to-tapd` 技能，开始阶段四的工作，无需等待用户再次触发。

### 阶段四：需求拆分与 TAPD 单据创建

- **技能**：`prd-to-tapd`
- **输入**：阶段三产出的标准 PRD + HTML 原型 + 审核报告
- **产出**：
  - iWiki 文档目录（包含 PRD、原型说明、审核报告）
  - TAPD 父需求 + N 个子需求
  - 父需求评论（引用 iWiki 链接）

执行流程：
1. **iWiki 上传**：创建功能目录，依次上传 PRD、原型、审核报告
2. **创建父需求**：从 PRD 提取整体信息，创建父级 TAPD 需求
3. **创建子需求**：PRD「功能详细设计」中每个 `### 3.x` 章节对应一个子需求
4. **添加评论**：在父需求上添加评论，引用所有 iWiki 文档链接

拆分规则：优先级从标题标签映射（P0→High, P1→Middle, P2→Low）。

**阶段四检查点清单（必须输出）**：

```markdown
## 检查点确认 - 阶段四：需求拆分与 TAPD 单据创建

### 本阶段产出
- [ ] iWiki 文档目录：https://iwiki.woa.com/pages/viewpage.action?pageId={目录ID}
  - [ ] 需求文档：https://iwiki.woa.com/pages/viewpage.action?pageId={PRD文档ID}
  - [ ] 交互原型：https://iwiki.woa.com/pages/viewpage.action?pageId={原型文档ID}
  - [ ] 审核报告：https://iwiki.woa.com/pages/viewpage.action?pageId={报告文档ID}
- [ ] TAPD 父需求：https://tapd.woa.com/tapd_fe/{workspace_id}/story/detail/{story_id}
- [ ] TAPD 子需求（共 N 个）：
  - [ ] 子需求 1：[模块名] (P0)
  - [ ] 子需求 2：[模块名] (P1)
  - ...
- [ ] 父需求评论已添加 iWiki 链接

### 待确认事项
1. 父需求和子需求的层级关系是否正确？
2. 子需求的优先级是否与 PRD 模块优先级一致？
3. 是否需要进入阶段五（研发闭环）？

### 下一步操作
如需继续，将进入 **阶段五：研发闭环**，基于 TAPD 单据进行开发实现。

---
请回复「确认」完成本轮需求流程，或回复「继续」进入阶段五。
```

### 阶段五：研发闭环

- **技能**：`bk-vibe-coding`
- **输入**：阶段四创建的 TAPD story
- **产出**：代码、测试、MR/PR、TAPD 评论

强制约束：所有代码变更必须绑定 TAPD story，禁止直推 master/main。

## 可用技能

| 技能 | 职责 | 所在阶段 |
|------|------|---------|
| product-design | 需求沟通 → HTML 原型生成 → PRD 撰写（串行三步） | 一、二、三 |
| prototype-capture | PRD 撰写时调用，对原型交互式截图并回填文档 | 三（子步骤） |
| prd-to-tapd | PRD → TAPD 父需求 + 子需求两层单据 | 四 |
| bk-vibe-coding | TAPD 单据驱动：状态流转 → 开发 → 提测 → 发布 → 闭环 | 五 |
