# 产品设计技能

覆盖从需求沟通到原型输出再到需求文档的完整产品设计流程。**三步串行执行**：先聊清楚，再出原型，最后补文档。

## 三步流程（串行执行）

### 第一步：需求沟通与确认

通过多轮对话引导用户将模糊的想法变成清晰的需求大纲。**本步骤只做沟通和梳理，不生成原型或文档。**

**支持的输入：** 设计稿、截图、Figma 链接、口头描述、一句话想法——任何形式均可

**沟通策略：**

| 维度 | 引导问题示例 |
|------|-------------|
| 业务背景 | "这个功能要解决什么问题？目前用户是怎么做的？" |
| 目标用户 | "谁会用这个功能？他们的角色是什么？" |
| 核心功能 | "必须实现的功能有哪些？哪些是后续可以迭代的？" |
| 页面结构 | "你希望这个页面大概分几个区域？主要展示什么信息？" |
| 关键交互 | "有哪些重要的操作？比如筛选、展开、跳转？" |
| 数据字段 | "需要展示哪些核心数据？有哪些关键字段？" |
| 边界约束 | "有什么技术限制、权限要求或性能考虑？" |

**沟通原则：**
- 每轮聚焦 2-3 个问题，不要一次全问
- 用户提供截图/设计稿时，先分析截图提炼需求理解，再针对性追问缺失信息
- 用户说得越少，越要主动引导；不确定的点直接问，不假设
- 每轮结束时总结已明确的内容和待确认的问题

**产出：需求大纲**（保存为 `docs/{功能名}_需求大纲.md`）

需求大纲包含：
- 功能名称
- 需求背景（一段话）
- 目标用户及角色
- 功能模块清单（附优先级 P0/P1/P2）
- 每个模块的核心能力描述（2-3 句话）
- 页面/模块结构
- 关键交互流程
- 已知约束和边界

**完成标准：** 输出需求大纲后，明确询问用户"这个需求大纲是否可以作为原型设计的基础？"。用户确认后方可进入第二步。

### 第二步：生成可执行 HTML 原型

基于已确认的需求大纲，输出**可直接在浏览器中打开运行的 HTML 原型文件**。

**原型要求：**

| 要求 | 说明 |
|------|------|
| 单文件可运行 | 输出为单个 `.html` 文件，内联 CSS 和 JS，浏览器直接打开即可预览 |
| 页面结构还原 | 按需求大纲中的页面结构设计布局、导航、表格、图表等 |
| 交互可操作 | Tab 切换、行展开/收起、按钮 hover、弹窗打开等核心交互可点击 |
| 数据可感知 | 使用 Mock 数据填充表格、图表、标签等，呈现真实使用效果 |
| 样式贴合设计 | 参考蓝鲸设计规范（`references/蓝鲸设计规范.md`），颜色、字体、间距尽量贴合 |

**输出物：**

| 产出 | 格式 | 说明 |
|------|------|------|
| HTML 原型文件 | `prototypes/{功能名}.html` | 可执行的交互原型，浏览器直接打开 |
| 功能结构图 | Mermaid | 功能模块的层级关系 |
| 操作流程图 | Mermaid | 用户核心操作路径 |

**执行流程：**
1. 基于需求大纲设计页面布局和交互方案
2. 生成可执行 HTML 原型文件，内联样式和脚本
3. 输出功能结构图和操作流程图
4. 引导用户在浏览器中预览原型，收集反馈并迭代
5. 用户确认后方可进入第三步

### 第三步：根据原型撰写 PRD

在原型确认后，基于 HTML 原型和需求大纲，撰写标准化的产品需求文档（PRD）。

**执行流程：**
1. 基于已确认的 HTML 原型，提取页面结构、字段、交互和状态
2. 按 PRD 模板撰写完整文档（参考 `references/prd-template.md`）
3. 撰写到"页面原型"章节时，调用 `prototype-capture` 技能对原型 HTML 进行交互式截图（根据 PRD 中定义的功能模块和交互状态决定截哪些内容），回填截图路径
4. 逐条检查撰写规范（参考 `references/prd-writing-guide.md`）
5. **必须输出审核报告**（参考 `references/review-report-template.md`）
6. 审核报告保存为独立文件：`[PRD文档名]_审核报告.md`
7. **输出检查点清单并等待用户确认**

> `prototype-capture` 是第三步的子步骤，不是独立阶段。PRD 撰写过程中明确了功能模块和交互状态后，才知道需要截取哪些页面、哪些状态的截图。

**串行约束：**
- 第三步依赖第二步（原型），第二步依赖第一步（需求大纲）
- 如果用户直接要求写 PRD，先检查是否有原型 → 是否有需求大纲，缺哪步补哪步
- 如果用户提供了充分的截图/设计稿且需求已经非常明确，第一步的沟通可以简化但不可跳过——至少输出需求大纲并获得用户确认

**自动衔接下一阶段（重要）**：

PRD 和审核报告完成后，必须输出检查点清单。用户确认后，**立即自动调用 `prd-to-tapd` 技能**，执行以下操作：
1. 在 iWiki 创建功能目录，上传 PRD、HTML 原型、审核报告
2. 创建 TAPD 父需求和子需求
3. 在父需求上添加评论，引用 iWiki 文档链接

**禁止在第三步结束后停止**——用户确认检查点后，必须继续执行 `prd-to-tapd` 技能。

## 最佳实践

1. **提供清晰的设计稿** — 确保图片清晰可读，确认页面所处产品结构层级
2. **详细描述需求** — 提供业务场景和用法介绍
3. **明确目标用户** — 说明产品的目标用户群体
4. **迭代优化** — 根据生成结果提供反馈，持续优化

## 输出规范

- 所有文档使用 Markdown 格式
- 图表优先使用 Mermaid，节点不可重叠
- 截图使用 `{序号}-{功能模块}-{子功能/状态}.png` 命名
- PRD 完成后必须附带审核报告
